Event Planner

A planning hub for the hosts of small gatherings.

A lightweight Notion workspace plus Google Form intake that replaces the group-chat + spreadsheet + sticky-notes coordination loop that most private hosts run on today.

See a live event page Read the PRD
The problem

Hosting a 12-person dinner takes 5 hours of logistics across 5 apps.

People hosting small gatherings of 5–50 guests coordinate across a group chat for invites, a spreadsheet for potluck, a notes app for the run-of-show, and texts for dietary restrictions. Information gets lost, duplicated, or forgotten, and the host carries all the cognitive load.

Existing tools either over-solve the problem (Eventbrite, Partiful, built for ticketed and public events) or under-solve it (group chats and plain spreadsheets with no structure). There’s no lightweight, structured planning hub aimed at the private host.

The solution

One workspace for the host. One form for the guests. One reminder before the day.

📋

Notion workspace

Five linked databases (Events, Guests, RSVPs, Potluck, Schedule) where the host plans everything. Relations and rollups give auto headcount, dietary roll-ups, and a past-events library for free.

🍽

Google Form intake

Guests RSVP and claim potluck slots without ever signing into Notion. Responses land in a Sheet that the host copies into Notion in v1, with Zapier auto-sync planned for v2.

📧

Day-before reminder

A Google Apps Script trigger sends each confirmed guest a single reminder with the address, time, what they signed up to bring, and the dress code. No app install required.

Built as a PM portfolio piece

What this project demonstrates.

The app itself is small on purpose. The repository is the real artifact. Every doc was written before code or template work began.

PRD

Problem framing

Problem statement, target users, jobs-to-be-done, and competitive positioning against Eventbrite, Partiful, and the group-chat default.

 MoSCoW

Scope discipline

Every feature classified Must / Should / Could / Won’t. Explicit out-of-scope list (payments, mobile, ticketing) with the reasoning behind each cut.

 Schema

System design

Five-database schema with relations, rollups, and formula properties. Designed so the host gets headcount and dietary totals without doing the math.

 Metrics

Success metrics

North-star (planning time per event), activation, retention, and guardrail metrics. Targets calibrated against a ~5-hour baseline.

 Roadmap

Iteration plan

Now / Next / Later with an explicit won’t-do column. V2 features (budget tracker, auto-reminder, Form auto-sync) are gated on pilot retro findings.

 Repo

End-to-end ownership

Public repo, public landing page, and planned real-world pilots with written retros published in-repo.
